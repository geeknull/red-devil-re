# 行为测试网 设计文档

> 日期：2026-07-03 · 分支：`feature/behavior-net`
> 依据：四路并行代码扫描 + 交叉核对（含 6 处源码实核、Node 环境部分实测）。
> 相关记忆：[[readability-refactor-state]]、[[fidelity-gaps]]、[[red-devil-decompile]]。

## 1. 背景与动机

red-devil-re 是两款 J2ME 游戏（红魔特种兵 1/2）的 TypeScript 忠实移植。可读性改造 P0（枚举/常量/JSDoc，同值替换）已完成提交。P1/P2（拆巨型方法、局部改名、收紧类型）**会改动代码结构与控制流，"还原成数值后逐字节对比 git" 的证明方法不再适用**。

仓库自身有前车之鉴：静态闸门（tsc/build）曾集体假绿，语义化改名导致敌人/Boss 静默消失（见 `docs/复盘-语义化改名静默崩溃.md`、`docs/04-验证方法-位级一致.md:47,64`）。**结论：动结构前必须先建运行期行为回归网。**

## 2. 职责边界

- **测试网只回答一个问题**：重构前后，游戏逐帧行为是否**位级不变**。
- **不背书语义正确性**——那由 CFR 权威源（`reverse/game{1,2}/2-decompiled-cfr/`，game1 6686 行 / game2 6358 行）+ docs 五件套人工把关。
- 因此黄金基准锁的是"当前 TS 港的**真实**行为"，不是"应该的行为"。这条边界决定了选项 A（锁自身行为）而非对拍原版 jar。

### 为什么不对拍原版 jar（FreeJ2ME-Plus）

- 原版 `setSeed(currentTimeMillis())` + 墙钟节流，种子/时钟不可固定；需 patch 模拟器 System 时钟或改字节码。
- 仓库自判周期非精确（`docs/实现进展与技术细节.md:20`、`docs/04:68`），精度封顶低于位级需求。
- GUI 无头抓帧零脚手架，估 1–2 周；GPLv3 隔离要求测试代码只能子进程黑盒调用，模拟器不入库需 CI 额外下载。
- **保留角色**：延续现状的人工关键帧对拍（Logo/主菜单/第1关首帧/Boss 战），不进自动网。

## 3. 四个已定决策

| 决策 | 选定 | 理由 |
|---|---|---|
| 快照载荷 | **绘制调用日志 + 状态字段快照 并录** | 绘制日志=位级主证（渲染是逻辑最终输出）；状态字段=可读副本。两层互补。 |
| 接缝边界 | **零源码改动**（全部驱动逻辑在 harness 侧） | 先把基准锁在当前真实行为上，避开"测试网建成前先改源码"的鸡生蛋。接缝重构留到网建成后。 |
| 运行器 | **零依赖自写**（`node --experimental-transform-types` + run-all 模式） | 与现有 `test:shim` 同风格、无供应链；每场景独立子进程正好解决静态单例污染。 |
| 首版范围 | **最小闭环先跑通** | 每游戏 3 场景验证全链路可复现，再扩。 |

一处设计侧自定（可否决）：**假时钟每帧 +frameStepMs（非冻结）**，令结算页"所用时间"文本与理想帧率真机一致。

## 4. 确定性现状（五类非确定源盘点）

| 来源 | 状态 | 处置 |
|---|---|---|
| **时钟 Date.now** | ❌ 唯一无注入缝 | 全局 monkeypatch 假时钟。散布三类：RNG 播种（`game1 GameMIDlet.ts:51` / `game2 GameMIDlet.ts:81`）、关卡计时（`GameScreen.ts:450,539,696` / `GameCanvas.ts:382,623`）、循环节流（手动驱动不经过）。关卡计时会进结算页**渲染文本**，故 stub 是硬需求。 |
| **RNG** | ✅ 有缝 | 单一全局静态 `GameMIDlet.random`（JDK 48 位 LCG 位级复刻，`Random.ts:15-40`），`setSeed` 公开。注入点 = `new GameMIDlet()` 之后、首帧 paint 之前。 |
| **输入** | ✅ 有缝 | `keyPressed/keyReleased` 公开（`GameScreen.ts:1663,1693` / `GameCanvas.ts:987,1008`），DOM 监听仅外围分发，可完全绕开。约束：game1 为 4 槽环形队列会环绕覆盖，每帧注入 ≤4 事件。 |
| **异步加载** | ✅ 全前置 | 唯一异步点在启动预加载；运行期取资源是同步 Map（`Resource.ts:176-179,194-198`）。 |
| **音频** | ✅ 天然确定 | 无 AudioContext 全链路 no-op（`Sound.ts:37-48,79`；game2 `GameMIDlet.ts:164-165,184-185`），不在覆盖面内，也无需 mock。 |
| **持久化** | ✅ 天然确定 | localStorage 有 typeof 守卫，Node 缺省走确定默认值（首次无存档）。存档字节应纳入初始夹具。 |
| **静态可变状态** | ⚠️ 约束 | `GameScreen.instance` + static 块、`LevelScene` 静态池（`:67-78`）、`TileMap.offscreenBuffer`（`:235`）→ **每用例独立进程**，不依赖手工 reset。 |

## 5. 架构与目录

新建 `packages/behavior-net/`，**游戏无关驱动核 + 每游戏适配器**：

```
packages/behavior-net/
  src/
    fake-ctx.ts        录制型 ctx（~21 成员：15 方法 + 6 属性），每次绘制记一条
    fake-canvas.ts     setCanvasFactory 注入的假离屏画布 + PNG IHDR 解析宽高
    fixtures.ts        fs 读 .bin → registerResource；注册 stub Image（含 game2 换色帧键 archive#index@frame）
    fake-clock.ts      monkeypatch Date.now，每帧 +frameStepMs
    driver.ts          游戏无关逐帧循环：注入输入→paint→录制→推进时钟→抓快照
    adapter.ts         GameAdapter 接口定义
    game1-adapter.ts   176×208 / 100ms/帧 / 先导 ProjectileActor / 杀 loopThread / 抽字段
    game2-adapter.ts   176×204 / 80ms/帧 / 杀 mainThread / 抽字段
  scenarios/*.ts       输入脚本（帧号→keyCode 序列）
  golden/*.json        黄金快照（每场景一份）
  run.ts               子进程隔离逐场景跑 + 对比 + 退出码聚合（照搬 run-all.ts）
  bless.ts             重录黄金（--bless）
```

### GameAdapter 接口（草案）

```ts
interface GameAdapter {
  screenSize: { w: number; h: number };   // game1 176×208 / game2 176×204
  frameStepMs: number;                     // game1 100 / game2 80
  bootstrap(seed: number): { screen: unknown };  // 见 §6 契约
  injectKey(screen: unknown, keyCode: number, down: boolean): void;
  paint(screen: unknown, g: unknown): void;      // screen.paint(g)
  snapshotState(screen: unknown): Record<string, unknown>;
}
```

## 6. 驱动契约（最载荷部分，全在 harness 侧，源码零改动）

顺序敏感，任一步错 game1 会**静默跑错**。⚠️ **`configureScreen` 必须在 `new GameMIDlet()` 之前**——Canvas 构造函数在构造时就从模块级 `SCREEN` 读 `getWidth()/getHeight()`（`Runtime.ts:69-74`），且 game1 `GameScreen` 构造期用它算 `screenWidth` / `playHeight`（`GameScreen.ts:150-151`）。

1. `globalThis.window = { addEventListener(){} }`（configureScreen 里挂键盘监听需要）。
2. `setCanvasFactory(假画布工厂)` + 装 `fake-clock`（monkeypatch `Date.now`）——**必须在 `new GameMIDlet` 之前**，否则 RNG 播种吃真 Date.now，且假画布工厂要先于任何 Image 创建就位。
3. game1 专属：**先 `import ProjectileActor.ts` 再 import index**，规避 `ActorBase.ts:25 ↔ ProjectileActor.ts:16` 循环导入 TDZ（`Cannot access 'ActorBase' before initialization`）。脆弱契约，入口文件固化 + 注释，动 import 结构前不可删。
4. `configureScreen({ width, height, scale:1, screen: 假 screen 对象 })`——设定模块级 `SCREEN`。game1 176×208 / game2 176×204。`Runtime.ts:36-38` 对 `screen` 只做 `.width/.height` 属性赋值，普通对象即可；跳过它 game1 会以 `screenWidth=0 / playHeight=-32` 静默跑错。
5. `fixtures` 全量注册资源——**首帧 paint 前必须齐**。Logo/Splash 分支逐帧取图走 `getCachedImage`，未注册即 throw（`Resource.ts:196`）；但两游戏 paint 有整体 try/catch 吞异常（`GameScreen.ts:225` / `GameCanvas.ts:189`）→ 缺资源不炸而是静默错帧。
6. `const midlet = new GameMIDlet()` → 立即杀构造期自启的常驻 setTimeout 循环：game1 `(midlet.screen as any).loopThread = null`（`run()` 的 while 判 `loopThread != null`，`GameScreen.ts:1569`）；game2 `midlet.canvas.running = false; midlet.canvas.mainThread = null`（`GameCanvas.ts:888`）。再 `GameMIDlet.random.setSeed(固定种子)`。**两游戏后台 `run()` 只调 `repaint()`，无 bindScreen 时 no-op、不 tick 逻辑（`Runtime.ts:95`），故杀循环只为让进程可退出、不影响状态。**
7. 逐帧：注入脚本输入（`keyPressed/keyReleased`，game1 每帧 ≤4 事件因 4 槽环形队列）→ `new Graphics(录制ctx, W, H)`（`Graphics.ts:40-47` 构造注入）→ 对 game1 调 `midlet.screen.paint(g)` / game2 调 `midlet.canvas.paint(g)` → 时钟 `+frameStepMs` → 抓快照。
8. 收尾 `process.exit(code)`（循环已杀，保险清 timer）。

### fake-ctx 已知两坑（实核）

- Graphics 构造即写 `ctx.imageSmoothingEnabled` / `ctx.textBaseline`（`Graphics.ts:45-46`）。
- `applyClip` 先 `restore()` 后 `save()`（`Graphics.ts:119-120`），**首次调用是不配对的 restore**，假 ctx 必须容忍。

### fake-canvas

- Image 离屏画布走官方缝 `setCanvasFactory`（`Image.ts:30-32`）。
- 假离屏 ctx 需 `createImageData/putImageData`（game1 RGB444 自解码管线 `GameScreen.ts:2528` 依赖）。游戏逻辑**从不回读像素**（全仓无 getImageData），故无需真解码；宽高从 PNG IHDR 纯字节解析即可（`TileMap.ts:348` 消费图集列数）。

## 7. 快照格式（两者并录）

每场景黄金文件 = 逐帧数组：

```json
{ "meta": { "game": 1, "seed": 12345, "frameStepMs": 100, "screen": "176x208" },
  "frames": [
    { "frame": 42,
      "drawHash": "<sha256 of 该帧规范化绘制调用序列>",
      "state": { "playerX": 0, "playerY": 0, "score": 0, "state": 10, "subState": 0,
                 "rngSeed": 0, "enemyCount": 0 } }
  ] }
```

- **drawHash**：位级主证。默认存哈希（紧凑）；不匹配时 harness 自动 dump 该帧完整绘制日志供人肉 diff。
- **state**：可读副本。字段清单取自 `docs/game{1,2}-*/玩法与数值.md`（`<<10` 定点坐标、分数、状态字、RNG 游标、关键 actor 坐标）。字段经 `as any` 读取（零源码改动）。
- 互补性：drawHash catch 纯渲染回归（state 漏掩的），state catch 逻辑回归且报告可读。
- **离屏画布按内容标识（实现要点，终审催生）**：位图字形/瓦片缓冲/game1 换色精灵先画进离屏缓冲再整块 blit。若离屏画布按「分配序号」标识，则 ① 缓冲内容变而 blit 位置不变→漏（误绿），② 行为中性的分配顺序/缓存复用→身份漂移（误红）。故离屏画布身份 = 其内 op 序列 + 尺寸的哈希，`putImageData` 并哈希像素字节。以「等长菜单文案改字」变异自检验证（旧序号方案会误绿，内容哈希方案变红）。

### 文本渲染的跨平台问题

不选像素级黄金帧：`ctx.fillText` 依赖系统字体（`Font.ts:55`），`drawRegion` 浮点旋转（`Graphics.ts:206-222`）跨 canvas 实现有 1px 差异。**绘制调用日志锚定的是参数序列（含 fillText 的字符串与坐标），跨平台确定**——这正是选调用日志而非像素比对的原因。

## 8. 错误处理与忠实性红线

- **静默吞异常 → drawHash 兜底**：源码零改动无法拆 paint 的 try/catch，但一帧中途抛异常会使绘制序列截断/改变 → drawHash 立即不匹配。这天然是 fail-fast 信号。fixtures 层再加"未注册资源即抛"断言防假绿。
- **忠实性硬约束**（`docs/可读性改造-backlog.md` 高危清单）：case fall-through、死代码守卫、`<<24>>24`/`<<16>>16` Java 截断、静默 catch——测试网只观测不改动，天然不破坏。

## 9. 首版范围（最小闭环）

每游戏 3 场景先跑通全链路：

1. `Logo → 菜单 → 第1关前 N 帧`（覆盖启动、资源加载、菜单导航、关卡进入）。
2. 一次`过关结算`（覆盖结算页 + 假时钟计时文本）。
3. 一次`死亡结算`（覆盖失败分支）。

game2 换色帧路径（最大保真缺口，见 [[fidelity-gaps]]）此版**不专门设场景**，但 fixtures 仍注册换色帧防 throw。跑通后再扩关卡/高危路径（fall-through、死代码守卫覆盖）。

**实现追加（超出首版最小闭环）**：终审指出菜单层场景不观测 actor（历史静默崩溃区），已补 `game1-level` / `game2-level` 两个**真进关卡**场景——game1 进 PLAYING（drawQueue 有 4 EnemyActor+3 BossActor 等逐帧模拟）、game2 进 InGame（场景含 EnemyActor/ItemActor/ProjectileActor）。以「两游戏 ActorBase 物理各加 1px/帧」变异自检验证：**仅两个 level 场景变红、菜单层保持绿**，证明 actor/敌人/Boss 模拟已被 drawHash 观测。故 §10 承诺对 actor 已兑现。

## 10. 验收：怎么证明测试网自己是对的

建成后做**变异测试自检**：故意在某 actor 逻辑里改一个常量（如敌人速度 +1），跑网必须**红**；改回必须**绿**。绿→红→绿 证明网真在观测行为——避免像 `ott.verify.ts`（`:12-17` 无断言恒绿）那样假通过。

## 12. 风险与脆弱契约

- **game1 循环导入顺序**：入口先导 `ProjectileActor.ts`，依赖 ESM 求值细节。必须固化 + 注释，动 import 结构前不可删。
- **运行器类型剥离**：必须 `--experimental-transform-types`（`Image.ts:35-41` 构造器参数属性 strip-only 不支持），与现有 `test:shim` 的 `--experimental-strip-types` 不一致，需在 run.ts 里统一。
- **假时钟每帧步进值**是固定契约，写进黄金文件头 `meta`。
- **每场景独立子进程**是硬约束，不可为省启动开销退回同进程复用。
