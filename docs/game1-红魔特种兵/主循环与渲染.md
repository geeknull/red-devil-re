## 1. 屏幕 / 坐标系

- 画布 `FullCanvas`（Nokia 全屏，无软键标签区）。构造时 `a = getWidth(); b = getHeight()-32`（`a.java:113-114`）：`a`=屏宽，`b`=游戏可视高（屏高减 32 像素 HUD 条）。N-Gage QD 物理屏 **176×208**，故 `a=176`，HUD 条占下方 `y∈[176,208)` 共 32px，游戏区 `176×176`（HUD 绘制用 `setClip(0,176,176,32)`，`a.java:961`）。
- 世界坐标定点：精灵坐标 `g.C/D` 为 **<<10 定点**（1024 = 1 像素，`g.java:29-30`）；速度/加速度同单位。屏幕坐标 = `(C>>10) - 相机`（`g.java:146-147`）。相机 `a.r/a.s` 在脚本中 <<10 与 <<14 混用（如 `r>>14`，`a.java:1085`）⚠️混用细节复刻时需逐处对齐。
- 像素缓冲 `J = new short[3600]`（`a.java:121`）：DirectGraphics `drawPixels` 用的临时行缓冲（i 类像素解码也有 `static short[4096]`，`i.java:32`）。

## 2. 主循环 `run()`（CFR：`a.java:1426-1444`）

```java
long t0 = System.currentTimeMillis();
while (this.aa != null) {                       // 线程引用非空即运行 (1429)
    if (this.X && !this.Y) {                    // 运行中且未挂起 (1430)
        long dt = currentTimeMillis() - t0;     // 本帧耗时 (1431)
        if (dt < this.W) {                      // W = 100ms (a.java:47)
            Thread.sleep(this.W - dt);          // 补足到 100ms (1433)
            System.gc();                        // 每帧 GC (1434)
        }
        repaint();                              // 触发 paint() (1436)
        t0 = currentTimeMillis();               // (1437)
    }
}
// 异常全部吞掉 (1441)
```

- **帧节奏：固定 100ms/帧 = 10 FPS**（`W=100L`，`a.java:47`）。不足睡补，超时不补（无追帧）。
- **暂停**：`Y=true` 时整循环空转（不更新不绘制）；`X=false` 同理。
- **逻辑更新位置（CFR 已确认）**：`run()` 本身只 `repaint()`，**逻辑更新 `a()`（`a.java:860`）由 `paint()` 在游戏内状态分支里、绘制 `a(Graphics)` 之前调用**。例如关卡内 `case 10: this.a(); this.a(var1_1);`（`a.java:469-472`）、`case 21:`（`a.java:446-447`）、`case 19:`（`a.java:713-714`）。相机更新 `c()`（`a.java:1073`）由 `a()` 在更新末尾调用（见 §3）。这是 J2ME 单线程「update+render 都在 paint」的典型写法。
- `paint()` 进入时置 `Y=true`、退出置 `Y=false`（`a.java:166`/`857`），与 `run()` 的挂起标志互锁；并 `++o`（`a.java:167`）作全局帧计数（菜单精灵动画相位 `o%3` 等用之，`a.java:225-226`）。整个 `switch` 包在 `try{}catch(Exception){}` 中，**绘制期异常被静默吞掉**（`a.java:856`）。

## 3. 每帧逻辑更新 `a()`（`a.java:860-949`）

每帧**重建渲染队列** `k[]`（容量 40，`a.java:1273`），顺序即绘制层序（后画的在上）：

1. 清空 `k[]`，`v=0`（866-871）。
2. 遍历当前屏块 `g.f[g.g]` 内的精灵 ID，激活且有效的：若 `b(q)`（属背景类）暂存 `nArray[]`（`a.java:865` 的临时 `int[10]`），否则压入 `k[]`（872-883）。
3. 非关卡4：压入玩家 `j`（884-886）。
4. `z>0` 时压入敌人矩阵 `m[2][3]`（及其关联特效 X）（887-901）；关卡4 压入 Boss `n` 及随从 `n.o`（902-909）。
5. 关卡4：玩家 `j` 在此压入（在敌人之上，911-913）。
6. 压入暂存的背景类精灵 `nArray[]`（914-918）。
7. 压入投射物池 `l[5][]` 中所有激活者（919-931）。
8. **物理更新**：对 `k[0..v)` 逐个 `e()`（932-936）。
9. **行为/AI**：再逐个 `a()`（937-941）。
10. `c()` 相机+脚本（942）。
11. 非关卡4 更新地图块流式加载 `g.b(camX,camY)`（945-947）；`f.a(camX,camY)` 地图层定位（948）。

> 两遍循环（先全 `e()` 再全 `a()`）保证同帧内所有物体先完成位移再统一判定交互，是确定性的关键，复刻须保持。

## 5. 双缓冲

- **地图层**自带离屏缓冲（`b.u`=`Image`、`b.x`=`Graphics`，`b.java:37/40`），随相机滚动**增量更新**（`b.a(Graphics,int,int,int,int)`，`b.java:171` 起），只重绘进入视口的条带，省 CPU。
- **整屏双缓冲**：`FullCanvas`/`repaint()` 由系统自动后台缓冲（J2ME 标准），代码未显式建第二 Image 做整屏 flip。

## 6. 复刻要点（TS+Canvas）

| 项 | 原版 | Canvas 对应 |
|---|---|---|
| 帧率 | 100ms 定帧 (a.java:47, W=100L) | `setTimeout`/`requestAnimationFrame` 节流到 10fps，dt<100 补齐 |
| 更新/渲染 | 同线程，paint 内 update→render | 一帧内先 `a()` 后绘制 |
| 像素绘制 | DirectGraphics 4444 + 调色板 | 预解码 PNG/ImageData，或运行时调色板 |
| 地图缓冲 | 离屏 Image 增量条带 | 离屏 canvas 同法或整图缓存 |
| 坐标定点 | <<10（1024=1px） | 整数定点保持位级一致，绘制时 `>>10` |
| 随机 | `GameMIDlet.a(int)` 用 java.util.Random | 复刻 48-bit LCG |
