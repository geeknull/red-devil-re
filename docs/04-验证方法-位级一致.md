# 验证方法 — 如何证明"位级/逐帧一致"

复刻是否"原汁原味"不能靠肉眼感觉，需分层、可复跑地证明。四层验证，逐层收紧。

## 第 1 层：确定性单元验证（已就绪）

对可完全确定的原语做参考序列断言：

| 验证 | 脚本 | 参考来源 | 状态 |
|---|---|---|---|
| `Random` 48bit LCG | `packages/j2me-shim/test/random.verify.ts` | 参考序列硬编码于该文件 `CASES`（由 JDK 17 `java.util.Random` 输出抄录） | ✅ 通过 |
| `.bin` 归档解析 | `packages/j2me-shim/test/resource.verify.ts` | `tools/extract_bin.py` 结果 | ✅ 通过 |

运行：
```bash
node --experimental-strip-types packages/j2me-shim/test/random.verify.ts
node --experimental-strip-types packages/j2me-shim/test/resource.verify.ts
```

> Random 是位级一致的命门：原版每次取随机数都经它，序列一旦偏差，敌人生成、掉落、Boss
> 弹幕全部发散。已对 seed=0/1/42/大数 各断言 8 个 `nextInt()`、`nextInt(100)`、`absMod(100)`。

## 第 2 层：资源字节平价

解码出的资产必须与原 JAR 内嵌字节完全一致：
- PNG：提取的 `images/*.png` 与归档内字节逐字节相等（解包器零拷贝切片，天然一致）。
- 自定义二进制（精灵帧/地图）：TS 解码器解析结果与 Python 解码器（`tools/`）交叉比对，字段、数量、尺寸一致。

## 第 3 层：逻辑平价（移植策略保证）

采用"兼容层 + 逐行移植"而非重写：
- 上层逻辑逐行对应 CFR 反编译源码（`reverse/gameN/2-decompiled-cfr/`）。
- 底层原语（第 1 层）已字节一致。
- ⇒ 行为天然一致。关键路径加**状态快照断言**：固定 seed 跑 N 帧，记录玩家/敌人坐标、分数、状态机值，回归比对。

## 第 4 层：画面/逐帧平价（外部依赖）

终极对照——与真机/模拟器逐帧比像素：
1. 用开源 J2ME 模拟器跑原 JAR：
   - 首选 **FreeJ2ME-Plus v1.52**（开源，Java/AWT GUI，macOS 可跑；下载与版本见 [`../tools/README.md`](../tools/README.md)）。
   - 备选 microemulator。
2. 在相同输入脚本下抓关键帧（Logo、主菜单、第 1 关首帧、Boss 战等）。
3. TS 复刻版以固定 seed + 相同输入脚本抓同样帧，逐像素 diff。

⚠️ **本机现状（2026-06）**：**模拟器已就位**——FreeJ2ME-Plus v1.52（jar 位于 `tools/emulator/`，不入库，见 [`../tools/README.md`](../tools/README.md)）已可在本机 Java 17 上运行，并成功加载本项目 MIDlet（`FreeJ2ME.log` 记录 `Loading MIDlet: Sina-RedDevil | Main Class: tjge.GameMIDlet`，Canvas 240×320，且生成了 `rms/` 存档）。**剩余工作**是 GUI 应用无头抓帧不便，尚未把"抓关键帧 + 逐像素 diff"自动化。故第 4 层目前仍以第 1+2+3 层三重验证（确定性原语 + 资源字节 + 逐方法忠实）+ 浏览器关键帧人工比对（已截图存档 `docs/screenshots/`）替代自动化像素终判。

> ⚠️ 三重验证只覆盖**静态层与对照层**，不保证"运行期端到端可玩"。本文档所属的整套静态闸门曾在一次语义化改名事故中**集体给出假绿灯**（两游戏静默崩溃却全绿），详见 [`06-复盘-语义化改名静默崩溃.md`](./06-复盘-语义化改名静默崩溃.md)。因此强位级一致须配合**真·人工实测或运行期断言**，不能仅凭静态闸门通过就宣告端到端收敛。

## 复跑与确定性

- 所有逻辑验证用**固定 seed**（覆盖原版 `setSeed(currentTimeMillis)`），使运行可复现。
- 输入用脚本化按键序列（帧号→keyCode），保证对照实验输入一致。
- 验证脚本全部可 `node --experimental-strip-types` 直接跑，无需安装依赖。

## 当前结论

**第 1、2 层已通过。第 3 层（逐方法对抗式校验）经四轮已收敛至干净：**
- 第 1 轮（全量 25 类通览）：发现 4 处偏差（1 HIGH 字节截断、2 MEDIUM 构造顺序等），全部修复。
- 第 2 轮（16 高风险类聚焦"位宽/符号/溢出"）：发现 7 处同类截断偏差（`(byte)/(short)` 取负缺 i2b/i2s 截断），逐处补齐 i2b/i2s 移位或掩码截断守卫后全部修复。
- 第 3 轮（8 个最大复杂类逐方法深读，覆盖 184 个方法）：**0 处偏差**。
- 第 4 轮（其余 17 个类逐方法深读，覆盖 ~127 个方法）：**0 处偏差**——至此**全部 25 个类均完成逐方法深度核对，约 311 个方法零新发现**（loop-until-dry 收敛）。
- 每轮修复后两游戏 `tsc` 零错误 + 浏览器关键帧人工比对未见差异。

> ⚠️ 关键帧比对只能捕捉首帧/特定帧的渲染差异，**不代表整局可玩性**；此类静态/关键帧闸门曾在改名事故中给出假阳性（全绿但游戏静默崩溃，见 [`06-复盘-语义化改名静默崩溃.md`](./06-复盘-语义化改名静默崩溃.md)）。上述"收敛"结论仅就**可对照 CFR 源的逐方法静态核对**而言，运行期可玩性须以真·端到端实测为准。

**第 4 层**模拟器（FreeJ2ME-Plus v1.52）已就位、已能加载本项目 MIDlet，目前仍以关键帧人工比对替代自动化像素终判（截图存档于 `docs/screenshots/`），剩余工作是把 GUI 应用的无头抓帧 + 逐像素 diff 脚本化。

> 结论：在可对照 CFR 源的范围内，两游戏移植在**核心逻辑层**已达位级一致；列明的必要偏差/盲区：① 与真机/模拟器的逐像素终判尚未自动化（无周期精确模拟器，故自动像素 diff 未做）；② MIDI 音色受设备音源影响、ASCII 字距等非逻辑细节不强求逐像素；③ 静态闸门曾漏过一次运行期全功能崩溃（见 [`06-复盘-语义化改名静默崩溃.md`](./06-复盘-语义化改名静默崩溃.md)），故"位级一致"不等于"端到端可玩已验证"。剩余非逻辑项：MIDI 音色（设备 GM 多音色 vs 方波合成，平台必要偏差，详见 caveat ②）、游戏2 位图字体（当前用系统字体近似）。音频本身已接入并能发声（`packages/j2me-shim/src/Sound.ts` 实现 IMA-ADPCM 解码 / Nokia OTT 音调 / WAV 解码，`Midi.ts` 提供 `MidiSynth` 方波合成；`packages/web/src/main.ts:153` 注入共享 `AudioContext`，用户手势后 resume），不在剩余项之列。
