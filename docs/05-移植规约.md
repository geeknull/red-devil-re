# 移植规约（CFR 反编译源 → TypeScript）

> 目标：逐行忠实移植，**核心逻辑层位级一致**（必要偏差另行列明，如 MIDI 音色随设备、ASCII 字距、
> 无周期精确模拟器故自动像素 diff 未做，详见 `04-验证方法-位级一致.md`）。所有移植代码必须严格遵循本规约，
> 以保证多人/多代理并行移植时结果一致、可编译、可对照。权威反编译源为 **CFR**：`reverse/gameN/2-decompiled-cfr/tjge/`。

> ⚠️ **更新（移植后）**：本规约描述的是"**沿用混淆名 + porting-contract 短名消歧**"的**初版移植约定**。
> 移植完成后，TS port 已用 ts-morph **语义化机械改名为友好名**（类 `a→GameScreen`、方法 `h_I→keyCodeToAction`、
> 字段 `q→heldKeyAction` 等，逻辑逐字节不变、已端到端验证）。**故 `packages/` 现为友好名**；本文出现的混淆名/短名
> 仅反映"与 CFR 的逐行对照关系"，不再是 `packages/` 的实际标识符。旧↔新映射见 `tools/rename-map.game{1,2}.json`，
> 改名工具 `tools/build_rename_map.cjs`+`apply_rename.cjs`。二次开发直接读 `packages/` 友好名代码即可。

## 1. 类与文件

- 每个 Java 类移植为同名 `.ts` 文件：`packages/gameN/src/<类名>.ts`，导出 `export class <类名>`。
- **（初版约定）类名沿用原混淆名**（`GameMIDlet`、`a`~`l`），最大化与反编译源逐行对照。
  ※ **现状**：移植完成后 TS port 的 类/方法/字段 与**源文件名**均已语义化为友好名（如 `GameScreen.ts` 内 `export class GameScreen`，文件名与类名一致，仅框架类 `GameMIDlet` 保留原名）。详见顶部更新说明与 `tools/rename-map.game{1,2}.json`；二次开发以 `packages/` 友好名为准，CFR 基准 `2-decompiled-cfr/` 仍用混淆名 `a`~`l` 作对照。
- 文件头注释写明：原类职责、对应反编译文件、可读别名（见各游戏 `类清单与职责.md`：`docs/game1-红魔特种兵/类清单与职责.md`、`docs/game2-深海战舰/类清单与职责.md`）。

## 2. 方法命名（解决 Java 重载）—— 强制查契约表

TS 不支持方法重载，故按 `reverse/gameN/porting-naming/porting-contract.json` 的确定性映射重命名：

- **框架/生命周期方法保留原名**：`paint` `keyPressed` `keyReleased` `showNotify` `hideNotify` `run` `startApp` `pauseApp` `destroyApp`，以及构造函数 `constructor`。
- **其余方法** = `原名 + '_' + 参数类型码`（同 `(名,描述符)` 全局映射到同一 TS 名，保证继承重写一致）：
  - 基本类型：`I`=int `J`=long `Z`=boolean `Y`=byte `C`=char `F`=float `D`=double `H`=short
  - 数组：每维前缀 `A`（如 `byte[]`→`AY`）
  - 对象：`G`=Graphics `M`=Image `S`=String `In`=InputStream `Rnd`=Random `T`+字母=tjge.x；其余取末段全名
  - 示例：`a()`→`a_`，`a(int)`→`a_I`，`a(Graphics)`→`a_G`，`a(int,tjge.d)`→`a_ITd`，`a(Graphics,int,int)`→`a_GII`，`a(int,int,int,byte[],boolean)`→`a_IIIAYZ`
- 调用点必须用契约表中的目标名。**移植任一类前先查该类在 contract.json 中的方法映射**。

## 3. 字段

- 字段保留原名（字段不重载）。Java 允许字段与方法同名（如静态字段 `a` 与方法 `a()`）；TS 中因方法已加后缀，二者不冲突。
- 静态字段 → `static`；实例字段在构造或声明处初始化。
- 类型标注：`int/short/byte/char`→`number`；`long`→视用法用 `number`（不溢出处）或 `bigint`（需 64 位处，逐一判断并注释）；`boolean`→`boolean`；数组→`Int32Array`/`Uint8Array`/`(T|null)[]`。

## 4. 数值语义（位级一致关键）

- **定点坐标**：原版坐标/速度用 `<< 10`（×1024）。移植保持 `<< 10` / `>> 10` 原样。
- **整数除法**：Java `int/int` 截断 → TS 用 `(x / y) | 0`（负数也截向零，与 Java 一致）。
- **取模**：Java `%` 对负数结果符号同被除数，TS `%` 同语义，直接用。
- **移位**：算术右移用 `>>`，逻辑右移用 `>>>`，与 Java 严格对应；左移结果用 `| 0` 保持 32 位。
- **溢出**：依赖 32 位回绕的运算，结果加 `| 0`。
- **随机数**：`GameMIDlet.a(n)` 原样保留为 GameMIDlet 静态方法（game1 `nextRandomMod`、game2 `randomBelow`），内部调用 `Random.nextInt()` 后取绝对值再 `% n`；shim 另提供等价的 `Random.absMod(n)` 备用，但当前两游戏均未调用。`Random.nextInt()/nextInt(n)` 直接对应（shim 已实现两重载）。
- **char 运算**：`(char)` 转换 → `& 0xffff`。

## 5. 运行模型

- `a implements Runnable`，`run()` 内 `while(true){ ...; Thread.sleep(t); }` →
  移植为 `async run()` + `await Thread.sleep(t)`（仅加 async/await，控制流逐帧一致）。
- `repaint()` 调 shim `Canvas.repaint()`（同步绘制到离屏缓冲再缩放 blit）。
- 资源：PNG 在预加载阶段异步解码，`a.f_I(i)`（image.bin 加载）改为从预解码表按索引同步取 `Image`；
  其余 `.bin`（精灵帧/地图/字符串）按字节用 `ByteReader`/`ResourceArchive` 同步解析（逻辑逐行移植）。
- 键码：`keyPressed(keyCode)` 收到 shim 映射后的 J2ME keyCode；本两款游戏直接用 `keyCode→动作` 映射（如 game1 `keyCodeToAction`），未用 `getGameAction`。如需 `getGameAction`，shim 已导出同名函数备用。

## 6. 第三方/平台类映射

| Java | TS |
|---|---|
| `java.util.Random` | `Random`（shim） |
| `javax.microedition.lcdui.Graphics/Image/Canvas` | shim 同名 |
| `javax.microedition.midlet.MIDlet`、`Display` | shim 同名 |
| `com.nokia.mid.sound.Sound` | shim `Sound` |
| `com.nokia.mid.ui.DirectGraphics` | shim `Image.createRGBImage` + Graphics（见游戏1 像素管线） |
| `System.gc()` | 删除（无意义，注释标注原位置） |
| `System.currentTimeMillis()` | `Date.now()`（仅用于种子/计时，注释标注） |
| `Math.abs/min/max` | `Math.abs/min/max`（注意整数语义） |
| `try/catch(Exception)` | `try/catch`，吞异常处保持空 catch |

## 7. 校验

- 每移植一类，对照 CFR 源**方法逐一核对**：方法集、控制流、常量、位运算一致。
- 关键路径加固定 seed 的状态快照断言（见 `04-验证方法-位级一致.md` 第 3 层）。
- 全量 `tsc --noEmit` 通过；浏览器中关键帧人工比对（本机已有 J2ME 模拟器（FreeJ2ME-Plus v1.52）但无周期精确模拟器，且 GUI 应用无头抓帧尚未脚本化，故自动化逐像素 diff 暂缺，详见 `04-验证方法-位级一致.md` 第 4 层）。
