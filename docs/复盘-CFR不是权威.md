## 1. 摘要（TL;DR）

真字节码 vs CFR 产出：

| | 语义 |
|---|---|
| **真字节码**（`javap tjge.l.f()` 214-237） | `n4 /= n;  this.H = this.D > n3 ? -n4 : n4;`（除法**无条件**、在分支之前） |
| **CFR 产出（错）** | `this.H = this.D > n3 ? -n4 : (n4 /= n);`（把除法**沉进了 false 分支**） |

算术印证：`29696/26 = 1142` → 真机 `-1142`；照抄 CFR → `-29696`。

**但本复盘的重点不是这个 bug，是它为什么能活这么久。**

---

## 2. 核心结论：整棵验证树只有一个根，而这个根从未被验证

把项目的每一道闸门列出来，看它**锚在哪**：

| 闸门 | 拿什么 vs 什么 | 能抓什么 | **抓不到什么** |
|---|---|---|---|
| 人工逐行移植 | port ↔ **CFR** | port≠CFR | **CFR≠字节码** |
| 人工审查 / N-agent 对抗验证 | port ↔ **CFR** | 同上 | 同上 |
| `behavior-net` golden | port ↔ **port(旧)** | 新改动破坏旧行为 | **旧行为本来就错** |
| `test:shim` | shim 原语 ↔ 硬编码 JDK 值 | 原语漂移 | 不碰游戏逻辑 |
| P1-3「机械内容归一自证」 | 重构后 ↔ 重构前 | 重构改变了行为 | **两边都错** |

**没有任何一道闸门锚在真实制品（`.class` 字节码）上。**
整棵树只有**一个信任根：CFR**——而它从未被验证过。下游全部继承它的错误。

### 2.1 最扎心的一点：再多测试也不可能发现

这**不是**「测得不够」：

- 多写 1000 个 behavior-net 场景 → **一个都发现不了**（golden 从 port 录，port 错则 golden 把错的当基准）
- 多做 10 轮人工审查 → **一个都发现不了**（审查对照的正是 CFR）

> **审查者与被审查者，用的是同一个错误参照。**

所以该缺陷不是「难找」，而是**在既有验证体系下结构性不可发现**。
出路不是「更努力」，是**换一个信任根**。

### 2.2 判据（可复用）

> 任何验证，先问：**从这道闸门到真实制品，隔了几跳？中间那个东西被验证过吗？**
>
> - 隔 **0 跳** → **绝对闸门**，能求真
> - 隔 **≥1 跳且中间物未验证** → **差分闸门**，只能证「没变」，**不能证「对」**

**「没变」的证明可以无限串联，而永远不接触真相。**

## 3. 为什么没人怀疑 CFR：三个机制

1. **「权威」标签自己产生豁免权。**
   文档与记忆白纸黑字写着「CFR 权威反编译基准」。**一旦某物被命名为权威，它就不再被检验**
   ——而这恰恰是它开始说谎的时刻。**标签干了它没资格干的认知工作。**

2. **「比替代品好」被悄悄升级成「正确」。**
   当年选 CFR 是因为它能恢复 jadx 失败的方法。`CFR > jadx` → 「基准」→ 「真值」。
   这三步跳跃没人喊停。

3. **可读性本身就是陷阱。**
   CFR 吐的是**排版整齐的 Java**，它*看起来*就是源代码。字节码长得像机器产物，你会本能地核对；
   而一份漂亮的 Java，你会本能地**相信**。
   > **一个有损的派生物，越可读，越危险**——它把自己伪装成了原件。

---

## 4. 诚实标注：本次改造者（我）也完整地在这个盲区里

不能只批评历史。**就在发现它的同一个 session 里，我自己一次都没怀疑过 CFR**：

- P1-3 全程写「对照真值：`reverse/game{1,2}/2-decompiled-cfr/`」
- 甚至**用 CFR 当裁决者去否决重构**（`readOneRleRow`/`parseSheetHeader`：「CFR 确认有差异，不抽」）
- 「机械内容归一 + break 位置计数 + N-agent 三方对抗」极其严格——**但严格地瞄准了错误的靶子**

即：**在极其认真地保住一个从未被验证过的参照的忠实度。**

> **保真的严谨 ≠ 求真的严谨。**
> 这个项目把全部纪律投在「别弄坏它」，几乎没有机器用来回答「它当初对吗」。
> 而前者*感觉*起来就像严谨——**这才是它最危险的地方。**

---

## 5. 为什么现在才发现

因为 `jvm-oracle` 是**第一个锚在真字节码上的验证**：信任根从「CFR」挪到了「真游戏」。

**值得注意**：建 oracle 的初衷是「把用户从人肉验收里解放出来」，**捞出缺陷是副产品**。
价值不来自「找得更努力」，而来自**换了根**。

## 6. 已做的收敛动作

1. **修复 + 锚定**：`ProjectileActor.computeHomingTrajectory` 改为无条件先除（行内附 javap 字节码证据，
   **防未来「照 CFR 改回去」**）；夹具 `packages/jvm-oracle/regress/game1-grenade-homing.txt`
   （由捞出它的 fuzz seed=110 固化）+ `diff.sh --script`。**已证夹具能抓旧缺陷**（回退→红且精确复现→恢复回绿）。
   ⚠️ 现有 golden **覆盖不到此路径**（既有场景只开武器0=导弹，不走 `computeHomingTrajectory`）
   → 修复后 golden 完全不变，**必须靠夹具锚定**。

2. **降级 CFR 的地位**（已同步 `docs/` 与记忆 `red-devil-decompile`）：
   - CFR 仍是**首选可读参照**（恢复 jadx 失败的方法，这点不变），但**不是最终裁决者**。
   - 遇到「port 与 oracle 分歧、但 port 看起来完全符合 CFR」→ **优先怀疑 CFR**，
     用 `javap -c -p` 裁决，**勿默认 port 错**。

3. **同类模式定向排查（已穷尽，4/4 裁决）**：
   高危签名 = **复合赋值/副作用混在三元里**。全仓扫描命中 4 处，逐个 javap 裁决：

   | 位置 | 写法 | 裁决 |
   |---|---|---|
   | `game1 l.java:156` | `H = D > n3 ? -n4 : (n4 /= n)` | 🔴 **CFR 错**（本次缺陷，已修） |
   | `game1 f.java:188` | `this.a = ab.x==4 ? (this.a &= -2) : 1` | ✅ 冗余但等价（真码为 if/else 各一次 putfield） |
   | `game2 GameMIDlet:105` | `n3 = ... ? (n3 \|= X) : (n3 \|= X&0xFF)` | ✅ 冗余但等价 |
   | `game2 a.java:730` | `this.m = K%2==0 ? (this.m -= 2048) : (this.m += 2048)` | ✅ 冗余但等价 |

   **精确化的危险签名**（#2/#3/#4 为何无害而 #1 有害）：
   > 三元里的副作用改的是变量 **X**，但三元结果赋给的是**另一个**变量，
   > **且另一支还读了 X 的未修改值** —— **这个非对称才是致命的**。
   （#2/#3/#4 中三元结果赋回的正是被改的那个变量，故冗余无害。）

   补扫 `++/--` 在三元、赋值嵌在三元分支：**0 命中**。

## 7. 认知边界（不可粉饰）

**模式扫描只能找到「我们已经知道长什么样」的错误。**

这个「高危模式」是**从那次发现里反推出来的**——**发现 #1 靠的不是模式，是模糊器；模式是事后总结的**。
CFR 完全可能有别的失效模式（循环条件重建、异常处理边界、switch 重构…），而**我们列不出来**。

> **所以「模式已扫净」≠「CFR 没别的错」。**
> 唯一能找到「我们不知道要找什么」的，是**以真字节码为根的差分模糊测试**。

### E/F 的落地情况（2026-07-16）

**F 已落地**：`pnpm verify`（`tools/verify-all.sh`）跑全部闸门，`regress/` 夹具**自动纳入**。
**已证能红**（退回手雷缺陷 → exit 1）。**但最有价值的是「谁抓到了它」**：`test:shim` 过、
`test:behavior` 过（**结构性抓不到**）、**连 `oracle:diff game1-level` 都过**（走不到手雷路径）——
**只有 `regress` 夹具红**。
> **「有绝对闸门」≠「被闸门覆盖」。闸门 × 覆盖，缺一不可。**

**E 部分落地**：
- **`game2 换色帧`：查明是[记载失实]，它一直被验着**（`game2-level` 586 条 drawRegion 与原版逐字节相同；
  变异测试证能抓旧缺陷）。**教训：错的不是代码，是标签**——与本复盘 §3.1 的「权威标签自产豁免权」同构，
  只是方向相反（那次让我们过度信任，这次让我们低估已有覆盖）。
- **模糊器改为状态覆盖引导**：game1 达 **11/15** 状态（新达 About/Help/Paused/**MissionFailed**/**CaptureCutscene**）、
  game2 达 **10/13**（新达 Help/About/**NoSave**）；两侧 **0 真分歧**。
  **未达**：game1 的 Ending/MissionComplete/GoalCutscene/LevelScroll、game2 的 LevelClear/LevelFailed/GameComplete
  —— 这些需要**打通关卡**，随机输入基本碰不到，**如实标注为未验证**。
- **新增 `coverage/*.cover` 覆盖断言**：把「覆盖了什么」从偶然变成机器断言的不变量（失败 exit 4）。
- 🔴 **副产品：捞出 oracle 自己的一个洞**——两侧不共享时钟，且**本复盘所依赖的审计文档原先断言
  「无需虚拟化时钟」是错的**（`javap`：`currentTimeMillis` 有 3 处在 `paint()` 内）。
  **失效模式与本复盘的又不同：前提对、推论错**——核实了「run() 改不改状态」，却没问
  「到底有没有东西读时钟」。**它能活这么久，同样是因为没有场景走到会显示耗时的界面。**
  详见 `jvm-oracle-保真审计.md`。

> ⚠️ **诚实标注**：本缺陷**已修复并实证**，但 E/F 待落地；且见 §7——**「模式已扫净」不等于「CFR 没别的错」**。
> 在 B（模糊测试）常态化跑出足够里程之前，**不要把「已修这一个」误读为「这类问题已清零」**。
