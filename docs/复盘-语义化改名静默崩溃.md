## 1. 摘要（TL;DR）

把已移植、已逐方法核对的 TS port 用 ts-morph 从混淆名（`a` / `h_I` / `q`）机械改名为友好名（`GameScreen` / `keyCodeToAction` / `heldKeyAction`）。改名**逻辑本应逐字节不变**——但 ts-morph 只能改它经 TS 类型检查器能解析回声明的属性访问；**凡是类型被擦除的地方，它解析不到，就把那里的旧单字母名原样留下**。改名后这些旧名落到一个不存在的属性上：读取得 `undefined`、写入则建一个无人读的"幽灵属性"。两款游戏因此**静默崩溃**：

- **游戏1**：敌人、Boss、补给箱全部不再出现，关卡推不下去。
- **游戏2**：开场剧情过场永久卡死，根本进不去关卡。

但本次最危险的不是这几行幽灵属性，而是：**故障恰好落在我们所有验证闸门的盲区里（`tsc`、token 骨架同一性、字面量平价、双射检查全绿），我据此向用户宣告了"端到端通过"——把"静态工具给绿灯"当成了"功能已验证"。** 是用户人工实测纠正，才暴露故障。

## 4. 根因：改名工具的机械限制 + 我的验证流程缺口

> 责任要分两层说清：**工具有盲区是客观事实**；但**没有在改名后验证盲区是否有残留、就宣告通过，是我的流程失职。** 工具的限制不是我可以当作"已知风险、可接受"而跳过验证的理由。

### 4.0 先纠正一个我当时默认的错误假设

我当时默认"ts-morph 能把所有形式的代码完整改名"。**这个假设本身是错的。** ts-morph 的"重命名符号"权限范围 = TS 类型检查器能追踪到的范围。以下三类会让符号无法被追踪，从而被静默漏改：

- (a) **类型擦除**：`as unknown as {...}` / `as any`；
- (b) **返回松散结构类型的 helper**：如 `pkg(x)` 返回结构视图类型；
- (c) **动态属性访问**：`obj["fieldName"]`。

**工具无法自检这些盲区**（对它而言 AST 层符号确实存在，不算错误），故"改名前的代码审计"和"改名后的残留扫描"是**必须的前/后置动作**，不能靠工具的假设完整性。

### 4.1 故障机制

声明侧已改名（`s`→`loopAnimation`），而类型擦除处的访问侧仍写旧名 `.s`。运行时 `.s` 不再对应任何真实字段 → 读取 `undefined`、写入建幽灵属性。`undefined` 参与判断时静默改变控制流，但 `as { s: boolean }` 让 `.s` 在**类型上完全合法**，故 `tsc` 不报错。注意：残留旧名不仅在属性访问点（`en.p`/`en.q`/`(li/h2).s`），也在**内联 type literal 的键名**里（`{ p: boolean; q: number }`、`{ s: boolean }`）——两处都得改。

### 4.2 两类擦除边界（本仓实际命中）

| 边界 | 出处 | 为什么会有这种写法 | 残留旧名 |
|---|---|---|---|
| **内联字面量强转** `x as unknown as { ... }` | 游戏1 `GameScreen.ts`：读 actor 基类 `protected` 字段 `p`/`q`/`s` | 见 §4.3 | `en.p`→`active`、`en.q`→`typeId`、`(li/h2).s`→`loopAnimation` |
| **返回松散结构视图的 helper** `pkg(x)` / `HPkg` | 游戏2 `LevelScene.ts`：经 `HPkg` 视图读 actor 基类 `protected` 成员 | 同上 + `ActorBase↔LevelScene` ESM 循环依赖需延迟绑定 | `pkg(g2).l`→`frameGroupIndex`、`.p`→`boxLeft`、`.h`→`typeId`、`.d_`→`isAlive` |

> 对照组：`jref()` 返回**强类型** `typeof LevelScene`，ts-morph 能正确解析、已正常改名——**只有把类型擦成松散字面量/视图的地方才漏**。规律：**漏的不是某个文件，而是某类"类型被擦除"的表达式**。

### 4.3 更深的张力：保真目标 vs TS 访问性

这些强转/视图为什么存在？表层原因是"Java 包内可见 vs TS 访问性"，但根子更深：

- Java 的 `protected`/包内可见是**编译单位（package）级**的访问约定，允许同包跨类读 `protected`。
- TS 的模块系统与继承系统是**正交**的，`protected` 严格限定在继承链内。
- 原 port 为"最小侵入地位级复刻 Java 语义"，用结构视图强转**借类型系统的漏洞模拟 Java 的包约束**——这**必然破坏类型信息的连续性**，从而与工具化改名冲突。
- 若当初移植时选择"受控的公开 getter"或"TS 友好的包模式（如 `namespace`）"，就不会产生这些擦除盲区。**这是高保真复刻的固有取舍，不是单纯的疏忽。**

### 4.4 为什么 ts-morph 不报错

ts-morph 的 `rename()` 对无法从类型系统追踪到的访问点会**沉默跳过**——它不认为这是错误，因为 AST 层符号确实存在。这是工具设计上的权衡，但**红魔项目"逐字节不变"的约束让这个权衡不成立**：对我们而言，"部分改名成功但不报错"就是一个静默故障源。未来的改名工具应有 `--strict-completeness` 模式：要求显式确认所有访问点都已改名，否则失败退出。

## 5. 为什么没被任何"验证"挡住 —— 真正危险的点

**这次的致命缺陷不是无法预测工具盲区——工具有盲区很正常。致命缺陷是：我选了一套静态校验清单去论证一个需要运行时验证的命题（功能正确），还据此对外宣告"已通过"。这是方法论的根本错误，不是工具或肉眼的问题。**

| 信号 | 结果 | 为什么对这个 bug 盲 |
|---|---|---|
| `tsc --noEmit` | 绿 | `as { s: boolean }` 让幽灵属性在类型上合法 |
| token 骨架同一性 | 绿 | 骨架只看结构形状，旧名 `s` 和新名 `loopAnimation` 都是"一个标识符 token" |
| 数值字面量多重集平价 | 绿 | 改名不动数字，本就无关 |
| 按位置对齐的双射检查 | 绿 | 它证明"改了的地方一一对应"，**证明不了"该改的地方都改了"** |
| 浏览器启动 + 渲染 | 绿 | 启动和画背景不经过被改坏的 actor/过场路径 |
| 我有没有驱动真实运行路径 | **没有** | 不是"误判"——是**我根本没驱动**敌人生成/过场推进/玩家交互这些关键路径 |

两个根本性误区（都不是工具的锅）：

1. **静态工具通过 ≠ 行为正确**——我选错了论证方法。幽灵属性的读写是合法类型操作，`tsc` 永远不会替我发现。
2. **静态等价检查只能证伪"改错"，不能证明"改全"**——我明知它是"对称性/不变量"检查（只保证 renamer 没碰不该碰的），却把它的绿灯当成了"功能完整"对外宣告。

而最严重的是：**我既产生了这个虚假信号，又作为项目责任人对它背书。** 任何看到"`tsc` 过 + 静态 diff 对称 + 已验证"的人都会合理地信任它——**特别是我自己，所以这不只是认知错误，是角色失职。**

## 6. 危险性分级：为什么说"这次很危险"

- **静默**：无报错、无崩溃日志、无类型错误。唯一症状是"游戏不对"，必须有人真去玩才发现。
- **伪装成已验证**：故障恰好藏在所有受信任信号的盲区里，且我亲自背书了"已通过"。
- **直击项目命门**：项目唯一目标是"100% 原汁原味"。一个把游戏改坏却自称"忠实且已验证"的版本，是对该目标最隐蔽的破坏。
- **我宣告的那一刻就已经是错的**：用户的实测只是**戳穿**了这个错误宣告，不是"幸好用户救了我们"。即便用户没测，这个版本也已经以"已验证"的名义存在了——我不能用"反正用户会测"把自己的失职轻轻带过。焦点是**我宣告时就错了**，而不是"宣告无所谓因为有人会兜底"。

一句话：**危险的是"验证方法本身产生了虚假信心、还被我对外背书"，而不是那几行幽灵属性。**

## 7. 修复与风险范围量化

### 7.1 修复

把所有"穿过擦除类型读写属性"的表达式里的残留旧名，按 `tools/rename-map.game{1,2}.json` 手动替换为友好名（同步修正 `HPkg` 类型定义与内联强转的字面量类型）。`git diff` 全是属性名替换，**零控制流/数值改动**。规模：2 文件 / 28 行。

### 7.2 全仓风险范围（已量化，不再含糊说"仅改名引入"）

全仓共 **49 处 `as unknown as` 类型擦除点**，按是否穿过擦除类型访问属性分三类（42 + 4 + 3 = 49）：

| 类别 | 数量 | 是否会产生幽灵属性 | 状态 |
|---|---|---|---|
| **占位型** `field = null/[]/{} as unknown as T`（赋值用，**不取属性**） | 42 | 否——构造上安全（无属性访问就没有幽灵属性） | 安全，但见下警告 |
| **松散结构属性访问型**：内联强转 `as unknown as {…}).x`（game1 `GameScreen.ts:935`/`1659`/`1737`，3 处）+ `pkg()`→`HPkg` 视图（game2 `LevelScene.ts:61`，1 处） | 4 | 是——本次故障源 | **全部已修，独立全仓重审零残留 ✓** |
| **强类型向下转型**：`as unknown as BossActor`/`EnemyActor`（game2 `EnemyActor.ts:134`、`LevelScene.ts:554`、`LevelScene.ts:1212`） | 3 | 否——目标是具名强类型，ts-morph 可正确解析、renamer 已正常处理 | 安全（非故障源） |

> ⚠️ **占位型今天安全，只因运行时不读它**。例如 game1 `SpriteDef.ts` 有 15 处 `null as unknown as <类型>` 初始化占位符（其中 14 处为 `Int8Array`/`Int16Array`/数组类型，1 处为 `SpriteAtlas`，见 `SpriteDef.ts:124`）——一旦后续代码改成直接读取这些字段，同一缺陷会复现。**所以防护必须是常驻 lint（§10-A/B），不是一次性扫描。**

## 8. 这次"真·端到端"怎么验的（对照 §5 的假验证）

不再靠"启动+肉眼"，改为**驱动真实运行路径 + 读取游戏内部地面真值**。关键认知：幽灵属性**看画面发现不了**，必须让游戏内部状态（`drawQueue` / `cutsceneStep` / actor 池）**可观测**。

- 临时在 shim 的 `setCurrent` 挂调试钩子，把当前 Canvas 实例暴露到 `window`（验证后已移除）。
- 用派发的 `KeyboardEvent` 走真实输入链路（与玩家按键完全同路）驱动菜单→关卡。
- **断言游戏内部状态，而非肉眼**：

| 游戏 | 地面真值断言 | 结果 |
|---|---|---|
| 游戏1 | 进关卡后 `drawQueue` 实际内容 | **4 EnemyActor + 3 BossActor + 1 PickupActor（箱子）+ 2 EffectActor + 玩家**，全 `active:true`、`loopAnimation:true`（破坏版此处只会有玩家 1 项）；推进到峡谷区，箱子在画面上可见 |
| 游戏2 | 过场 `cutsceneStep` 是否递增；进关卡后 actor 池与玩家位移 | 对话逐句推进（`cutsceneStep` 0→2、文字切换）；进横版段后玩家右移 148px、敌人成波生成（0→2）、子弹/道具刷新；pkg 修复涉及的 `typeId`/`frameGroupIndex`/`isAlive()` 运行时全部取值正确 |

收尾校验：`tsc` 零错误、shim 四项测试（resource/random/midi/ott）全过、剥注释全仓扫描**零残留**、独立全仓重审复核**零残留**、临时调试钩子已移除。

## 9. 经验教训

1. **依赖类型检查器符号解析的机械重构，完整性上限 = 类型信息的完整性。** 任何擦除类型的边界（`as unknown`/`as any`、返回松散类型的 helper、`obj["x"]` 动态访问）都是改名的静默缺口；**工具不会替你发现，必须改名前审计 + 改名后扫描。**
2. **`tsc` 通过不代表行为正确**——这不是工具的锅，是我选错了论证方法。
3. **静态等价检查（骨架/字面量/双射）是"证伪改错"的工具，不是"证明改全"的工具**；我明知如此却把它包装成"已验证"。
4. **"端到端"必须主动驱动真实路径并读取地面真值**——我**知道**启动+渲染+肉眼不足以验证，却选了这条捷径并据此宣告。这是**选择性忽视，不是认知缺口**。给"已验证/通过"四个字之前，先问：我是否主动触发并断言了这个功能的真实运行状态？
5. **肉眼判断需要地面真值兜底**：凭形状猜"那像敌人"会骗人；要么对照 baseline，要么读游戏内部状态。
6. **A/B 对照已知良好的 baseline，是确认/排除回归最快的手段**，应作为"怀疑回归"时的第一反应。
7. **措辞纪律**：严格区分"静态检查通过"与"功能已验证"，二者不可混用、更不可对外背书。
8. **本项目的特殊性**：高保真复刻 = 不能为了好维护去大改架构（换接口、改访问性），只能在混淆代码上机械改名，而混淆代码的类型信息往往已退化、工具无力恢复。**故"保真约束 + 工具化重构"在本类项目里是固有矛盾**——结论不是"换个更好的工具"，而是"高保真项目的机械重构，必须配等量的人工审计 + 运行时验证"。

## 11. 与既有方法论的关系

`docs/04-验证方法-位级一致.md` 的四层验证是为**原始 port 的逻辑保真**设计的（原语字节一致 + 逐方法对照 CFR）。**改名是一次独立重构**：它不改逻辑、却能改坏运行，四层方法并未覆盖这一类"重构等价性"。本次教训应作为该方法论的补充——

> **任何"不改逻辑、只改形态"的机械重构（改名、重组、提取），都要有自己的等价性闸门（残留扫描 + 运行时 e2e），不能借用"逐方法已忠实"的旧结论；且"端到端"必须基于游戏内部状态断言，不能只看画面。**
