# 反编译技术选型与决策记录

> **一句话结论：本项目两款游戏的权威反编译基准统一采用 CFR 0.152，弃用 jadx。**
> jadx 对本游戏的核心方法会**整段放弃（吐空壳）或输出"明知可能错"的代码**，CFR 则给出**完整可用的方法体**（个别方法即便 CFR 自标"未能完全结构化"，仍输出等价代码而非空壳，详见 §3.2）。
> 此文档为永久记录——**请勿再为本游戏考虑用 jadx 做反编译基准**。

---

## 1. 背景

两款游戏（`tjge` 包，game1=红魔特种兵 / game2=深海战舰）是 2000 年代的 J2ME 商业游戏，
**经过混淆**（类/方法/字段全为单字母 `a`~`l`）且**控制流不规则**。这种代码对反编译器是高难度场景。

权威真值是字节码本身：`reverse/<game>/1-jar-unpacked/tjge/*.class` + 原始 `.jar`。
反编译只是把字节码渲染成可读 Java——**不同工具的渲染质量差别极大**，本项目据此选型。

## 2. 工具对比（技术原理 + 实测）

| | **CFR 0.152** ✅ 采用 | **jadx 1.5.5** ❌ 弃用 |
|---|---|---|
| 内部原理 | 栈式、**以控制流重建为中心**，把任意跳转图强行结构化成 Java | 寄存器式 IR + **SSA + 类型推断 + 区域化**（Android/dex 出身） |
| 设计取向 | **务必完整出全**，畸形控制流也死磕到底 | 可读性/反混淆优先，**难啃的方法宁缺毋滥** |
| 遇到本游戏难方法 | 出全完整方法体（个别自标"未完全结构化"但仍是可编译等价代码，非空壳） | 某一遍假设崩 → **整方法放弃** |
| 自包含 | 零依赖单 jar（2.1MB） | 多模块 + 依赖 + GUI |

## 3. 实测铁证（采集自删除前的 jadx 产物）

### 3.1 jadx 默认：核心方法直接放弃，吐空壳

`game1` 主渲染/状态机 `paint()`（4710 条指令）在 jadx 默认输出里是：

```java
public final void paint(javax.microedition.lcdui.Graphics r13) {
    /*
        Method dump skipped, instruction units count: 4710
    */
    throw new UnsupportedOperationException("Method not decompiled: tjge.a.paint(...)");
}
```

**这是移植绕不开的核心逻辑，jadx 直接交白卷。** 同一个 `paint()`，CFR 输出了完整可用的方法体（`a.java:165` 起约 700 行，结构化代码而非空壳——尽管 CFR 在 `a.java:163` 自标"未能完全结构化"，详见 §3.2）。

### 3.2 失败规模统计

| 指标 | game1 | game2 |
|---|---|---|
| jadx 默认 — 含空壳方法的文件数 | **4**（a/f/h/k.java） | **5**（a/b/g/j/k.java） |
| jadx — `JADX WARN/ERROR` 标记数 | **23** | **47** |
| jadx `--show-bad-code`（full）— 仍带的失败/告警痕迹 | **23** | **47** |
| **CFR — 空壳数**（`Method dump skipped` / `UnsupportedOperationException`） | **0** | **0** |
| **CFR — 自标记 `Unable to fully structure code`** | **3**（a/e/f.java，见 `summary.txt`） | **0** |

> 说明：game1 那 3 处 CFR 自标记（`a.java` 的 `paint()`、`e.java` 的 `a()`、`f.java` 的 `h()`）只表示 CFR 承认该方法**未能完全结构化**——但它**仍输出了完整、可编译的等价方法体**（如 `paint()` 标记在 `a.java:163`、方法体从 `a.java:165` 起约 700 行真实代码），**不是 jadx 那种空壳交白卷**。两者性质完全不同。

### 3.3 行数对照（game1 `a.java`）

| 来源 | 行数 | 说明 |
|---|---|---|
| jadx 默认 | 1376 | `paint` 等是空壳 |
| jadx full（`--show-bad-code`） | 2192 | 空壳填成"**可能错**"的代码 |
| **CFR** | **2244** | **完整、结构化** |

> ⚠️ **jadx full 最坑**：它把空壳填成**看似正常、实则被 jadx 标记"可能错误"**的代码——
> 不细看会误以为是对的。`--show-bad-code` 这个参数名本身就是 jadx 在说"接下来是 bad code"。

## 4. 为什么 CFR 可信：可复现性已验证

- `2-decompiled-cfr/` 是 CFR 0.152 **默认参数直出**、未经人工改动（注释也是 CFR 自带的）。
- **独立来源复现**：用 Homebrew 官方 `cfr-decompiler 0.152`（与本机 `tools/decompilers/cfr.jar` 字节不同、SHA 不同）
  重新解码两款游戏 → 输出 **25/25 文件逐字节一致**。
- 证明：反编译是**确定性**的，`2-decompiled-cfr/` 可被任何 CFR 0.152 独立复现。

## 5. 为什么删掉 jadx 产物

1. **没有保留价值**：所有核心方法 CFR 已完整覆盖，jadx 是有空洞/可能错的劣质渲染。
2. **是隐患**：jadx full 的"bad code"看着像对的，留着容易被后人误当参考、引入错误。
3. **避免重复踩坑**：本文件即"已评估过、jadx 不适用"的结论，省得后人再去试一遍 jadx。

> 已删除：`reverse/<game>/decompiled_jadx/` 与 `reverse/<game>/decompiled_jadx_full/`（两游戏）。

## 6. 复现命令（如需重新生成 CFR 基准）

> **关于 cfr.jar**：CFR 0.152 是**第三方工具、不入库**（已从 Git 跟踪移除并 `.gitignore` 忽略，新克隆仓库不带此文件）。
> 需按 [`../tools/README.md`](../tools/README.md) 自行下载到 `tools/decompilers/cfr.jar`（CFR 0.152，下载地址见该 README）。

```bash
# 用本机 tools/decompilers/cfr.jar（按 tools/README.md 下载到位后，用本机 Java 即可）
java -jar tools/decompilers/cfr.jar "reverse/game1/0-jar/红魔特种兵.jar" --outputdir <out>
java -jar tools/decompilers/cfr.jar "reverse/game2/0-jar/红魔特种兵2-深海战舰.jar" --outputdir <out>
# 或 Homebrew 版（命令名是 cfr-decompiler，不是 cfr）
cfr-decompiler "reverse/game1/0-jar/红魔特种兵.jar" --outputdir <out>
```

## 7. 目录现状（reverse/<game>/）

| 目录 | 内容 | 地位 |
|---|---|---|
| `0-jar/` | 原始 jar 原物（`红魔特种兵.jar` / `红魔特种兵2-深海战舰.jar`，原名保留） | **原物（终极真值）** |
| `1-jar-unpacked/` | `0-jar` 的解包：`.class` 字节码 + `res/` 资源 | 真值（解包） |
| `2-decompiled-cfr/` | CFR 0.152 原始反编译输出 | **权威基准** |
| `3-readable/` | 可读层（自包含）：`tjge/*.java`（CFR 基础上人工重命名、逻辑逐字节等同）+ `SYMBOLS.md`/`glossary.json`（单字母→友好名 符号词典）+ `README.md` | 阅读用 |
| `porting-naming/` | 给 TS 移植命名（侧支）：`signatures/`（javap 签名）+ `porting-contract.json`（Java 重载→TS 名 消歧契约），**服务 `packages/` TS 移植**（与 3-readable 无关） | 移植用 |
| ~~`decompiled_jadx/` / `decompiled_jadx_full/`~~ | jadx 产物 | **已删除（劣质，见本文档）** |
